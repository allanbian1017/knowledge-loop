# RCA: Newsletter Raw Content Truncation

## 1. Observed Problem
The raw content included in today's newsletter reports (e.g., `Newsletter_2026_05_15/huggingface_Daily_papers_of_14_May_2026.md` and `Zhu_Qi_One_Fish_Multiple_Meals_Content_Reuse.md`) is unexpectedly short. In the Hugging Face report, only the first two papers are listed despite the original email containing 12+ papers.

## 2. Alternative Hypotheses & Rejections
- **Hypothesis 1: `gws gmail +read` is failing to parse the email body completely.**
  - **Evidence/Rejection**: Tested `gws gmail +read --id 19e27ba95c421074 --format json` on a different newsletter (Matt Pocock). The output contained the complete text. This proves the tool parses the text successfully when the text is provided by the sender.
- **Hypothesis 2: The LLM summarization agent is cutting off the raw content.**
  - **Evidence/Rejection**: The "Raw Content" appended to the Markdown reports perfectly matches the raw output generated by the `gws gmail +read` command. The LLM is faithfully reproducing the input it receives.
- **Hypothesis 3: All truncated reports are due to technical extraction errors.**
  - **Evidence/Rejection**: Checked the HTML body of `Zhu_Qi_One_Fish_Multiple_Meals_Content_Reuse.md` (`id: 19e248f2bc1554c6`). The `text/html` body is identical in text length to the `text/plain` body. The email is simply a short promotional teaser for a course, which is intended behavior by the sender.

## 3. Root Cause
- **Primary Root Cause**: The `gws gmail +read` tool defaults to returning the `text/plain` MIME part if it exists in a multipart email. Newsletter platforms (like Hugging Face) often heavily truncate the `text/plain` part, placing the complete article only within the `text/html` part. As a result, the ingestion pipeline only sees the truncated text.
- **Contributing Cause**: Some newsletters (e.g., Substack course promos) are legitimately short teasers linking to external sites, amplifying the perception of truncated content.

## 4. Fix Applied
- Modified `.agents/skills/ingest-newsletter/SKILL.md` to default to the `--html` flag when fetching emails to ensure the complete body is extracted.
- Modified `.agents/skills/content-summary/references/output_template.md` to explicitly instruct the agent **not** to append the "Raw Content" section for Newsletters, as the HTML-heavy format makes it too noisy and unnecessary in the final report.
